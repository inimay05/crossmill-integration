## 4. Missing Pieces

### 4.1 For HuggingFace deployment

- **`app/main.py`** in both `crossmill-safenutri/app/` and `crossmill-megaforge/app/`. Should be:
  ```python
  from openenv.core.env_server import create_app
  from app.environment import SafeNutriEnv  # or MegaForgeEnv
  from app.models import Action, Observation
  app = create_app(SafeNutriEnv, Action, Observation, env_name="crossmill-safenutri")
  ```
- **README front-matter** for both env Spaces (need `sdk: docker` + `app_port: 8000`, or build a custom Spaces Docker image). Neither repo's README declares Spaces metadata.
- **`Dockerfile`** for both env Spaces — none present.
- **`gradio` in `crossmill-integration/requirements.txt`** — currently missing (`pandas` and `huggingface_hub` only). Spaces auto-installs gradio because of the README spec, but a frozen pin is the right pattern.
- **`stable-baselines3`, `sb3-contrib`, `gymnasium`, `matplotlib`, `torch`, `numpy` in `crossmill-integration/requirements.txt`** — needed for `scripts/train.py` to run.

### 4.2 OpenEnv interface requirements not met

- `crossmill-megaforge/app/environment.py` extends `from openenv import Env` (a stub). Must extend `from openenv.core import Environment` (which provides `state`, `get_metadata`, `reset_async`, `step_async`).
- Both env `Observation` Pydantic models lack `reward: float | None`, `done: bool`, `metadata: dict` — required for `serialize_observation`. Either inherit from `openenv.core.env_server.types.Observation` or restructure the env to return those fields flat.
- Both env `StepResponse.reward` is a Pydantic `Reward(value, breakdown)` model. OpenEnv's `StepResponse.reward` is `float`. The internal breakdown should be moved into `info["reward_breakdown"]` and `reward` should be a flat float.
- `MegaForgeEnv.reset()` does not accept the `episode_id` kwarg that OpenEnv passes; SafeNutri does (`def reset(self, seed=None, episode_id=None, **kwargs)`).

## 7. Recommended Fix Order

1. **Unblock training (bugs #1, #2, #3)**: pick one of two strategies.
   - **Strategy A** — drop the `STRATEGY_DIM` concept entirely: set `AUGMENTED_OBS_DIM = {'safenutri': 23, 'megaforge': 26}`, remove `strategy_bias` concat from `gym_shim.reset()`, remove the `- STRATEGY_DIM` subtraction in `augmentation.py:95`, fix `LSTMPolicyAdapter` (already correct under this strategy).
   - **Strategy B** — keep strategy_bias: also concat `strategy_bias` in `gym_shim.step()` (currently only `reset()` does), and have `LSTMPolicyAdapter` append `np.zeros(STRATEGY_DIM)` to its `aug_vec` before `model.predict`.
2. **Unbreak megaforge over HTTP (bug #7)**: `from openenv.core import Environment` and either implement `state` / `get_metadata` / `reset_async` like SafeNutri does, or delete that import and use the SafeNutri pattern.
3. **Make `/reset` and `/step` return 200 on both envs (bug #8)**: the smallest change is to flatten `reward` to a float on the response and stash the breakdown into `info`. The bigger change (probably right long-term) is to inherit the OpenEnv `Observation` and `Action` base classes.
4. **Add `app/main.py` to both env repos (bug #6)**: 4 lines each.
5. **Move `theme` and `css` from `demo.launch(...)` to `gr.Blocks(...)` in `app.py` (bug #9)**.
6. **Fix `requirements.txt` for crossmill-integration (bug #10)**: pin `gradio==5.0.0`, plus the SB3 / matplotlib / torch / gymnasium stack if training is meant to run inside the Space.
7. The remaining medium / low items are quality-of-life cleanups.

## 8. Honest Bottom Line

The **logic** is impressively well-tested — 44 / 44 in the integration suite, 5 / 5 and 6 / 6 in the env suites. The memory architecture, grader anti-hacking rules, and the env physics all work in process. **None of the three repos are deployable as HF Spaces today**: the Gradio Space is closest (a one-line fix and a requirements update away), but the two OpenEnv FastAPI services are blocked by missing `app/main.py`, broken inheritance for MegaForge, and a non-conformant `Observation` / `Reward` contract for both. Training is also blocked by an obs-shape inconsistency that crashes on the first SB3 step. None of these are deep architectural problems — they look like late-stage glue bugs that surfaced when the strategy-bias feature was layered on top of an already-working memory pipeline.
